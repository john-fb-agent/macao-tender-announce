## 1. 系統概述

### 1.1 專案目標

建立一個自動化系統，定期監控澳門特別行政區公報網站的公開競投公告，當發現新公告時自動通知用戶。

### 1.2 核心需求

| 需求 | 說明 | 優先級 |
|------|------|--------|
| **自動抓取** | 定期從目標網站抓取公告資料 | P0 |
| **新公告檢測** | 準確識別新公告，避免重複 | P0 |
| **即時通知** | 發現新公告時立即通知用戶 | P0 |
| **歷史記錄** | 儲存所有公告，支援查詢和統計 | P1 |
| **穩定運行** | 7x24 小時穩定運行，錯誤自愈 | P1 |
| **易於維護** | 代碼清晰，文檔完整 | P2 |

### 1.3 非功能需求

- **可用性**：99% 以上
- **準確性**：公告檢測準確率 99% 以上
- **延遲**：新公告發布後 30 分鐘內通知
- **可維護性**：模組化設計，易於擴展

## 3. 模組設計

### 3.1 模組劃分

```
src/
├── main.py              # 主程式入口
├── crawler/
│   ├── __init__.py
│   ├── fetcher.py       # 網頁抓取
│   ├── parser.py        # HTML 解析
│   └── validator.py     # 資料驗證
├── detector/
│   ├── __init__.py
│   ├── comparator.py    # 新公告比對
│   └── deduplicator.py  # 去重處理
├── notifier/
│   ├── __init__.py
│   ├── telegram.py      # Telegram 通知
│   ├── email.py         # Email 通知
│   └── logger.py        # 日誌通知
├── storage/
│   ├── __init__.py
│   ├── database.py      # SQLite 操作
│   └── exporter.py      # 資料匯出
└── utils/
    ├── __init__.py
    ├── config.py        # 配置管理
    ├── logger.py        # 日誌工具
    └── helpers.py       # 輔助函數
```

### 3.2 模組職責

#### Crawler 模組

| 類別/函數 | 職責 |
|-----------|------|
| `Fetcher` | 發送 HTTP 請求，獲取網頁內容 |
| `Parser` | 解析 HTML，提取公告資料 |
| `Validator` | 驗證資料完整性 |

**流程**：
```
URL → Fetcher → HTML → Parser → Data → Validator → Validated Data
```

---

#### Detector 模組

| 類別/函數 | 職責 |
|-----------|------|
| `Comparator` | 比對新舊公告，識別新項目 |
| `Deduplicator` | 移除重複公告 |

**流程**：
```
New Data + Existing Data → Comparator → New Items → Deduplicator → Unique New Items
```

---

#### Notifier 模組

| 類別/函數 | 職責 |
|-----------|------|
| `TelegramNotifier` | 發送 Telegram 通知 |
| `EmailNotifier` | 發送 Email 通知 |
| `LogNotifier` | 寫入日誌檔案 |

**流程**：
```
New Items → Notifier → Formatted Message → User
```

---

#### Storage 模組

| 類別/函數 | 職責 |
|-----------|------|
| `Database` | SQLite 資料庫操作 |
| `Exporter` | 匯出為 CSV/JSON |

**流程**：
```
Data → Database → SQLite File
Data → Exporter → CSV/JSON File
```

## 5. 技術選型

### 5.1 程式語言

| 選項 | 優點 | 缺點 | 選擇 |
|------|------|------|------|
| **Python 3.8+** | 生態豐富，開發快速 | 執行速度較慢 | ✅ |
| Node.js | 非同步 IO 優勢 | HTML 解析生態較弱 | ❌ |
| Go | 執行速度快 | 開發效率較低 | ❌ |

**決策**：選擇 Python，因為：
- BeautifulSoup/lxml 生態成熟
- 快速原型開發
- 足夠的效能應付需求

---

### 5.2 網頁爬蟲

| 選項 | 優點 | 缺點 | 選擇 |
|------|------|------|------|
| **requests + BeautifulSoup** | 簡單直接，依賴少 | 不支援 JavaScript | ✅ |
| **Scrapy** | 功能強大，擴展性好 | 較複雜，過度設計 | ❌ |
| **Selenium/Playwright** | 支援 JavaScript 渲染 | 資源消耗大，速度慢 | ❌ |

**決策**：選擇 requests + BeautifulSoup，因為：
- 目標網站是傳統 HTML，無需 JavaScript 渲染
- 簡單易用，維護成本低
- 效能足夠

---

### 5.3 資料庫

| 選項 | 優點 | 缺點 | 選擇 |
|------|------|------|------|
| **SQLite** | 輕量，無需伺服器 | 不支援高併發 | ✅ |
| PostgreSQL | 功能強大，支援高併發 | 需要額外伺服器 | ❌ |
| MongoDB | 靈活，易擴展 | 需要額外服務，過度設計 | ❌ |

**決策**：選擇 SQLite，因為：
- 單一伺服器部署
- 資料量小（預計每年數千筆）
- 無需併發寫入

---

### 5.4 通知方式

| 選項 | 優點 | 缺點 | 選擇 |
|------|------|------|------|
| **Telegram Bot** | 即時，免費，易整合 | 需要網路 | ✅ |
| **Email** | 正式，可追蹤 | 延遲，可能被歸類為垃圾郵件 | ✅ |
| **Webhook** | 靈活，可整合其他系統 | 需要額外開發 | ❌ |

**決策**：支援 Telegram 和 Email，因為：
- Telegram 即時通知
- Email 作為備份和正式記錄

---

### 5.5 定期執行

| 選項 | 優點 | 缺點 | 選擇 |
|------|------|------|------|
| **Cron** | 系統內建，穩定 | 配置較複雜 | ✅ |
| **APScheduler** | Python 內建，靈活 | 需要程序常駐 | ❌ |
| **Heartbeat (OpenClaw)** | 與現有系統整合 | 依賴 OpenClaw | ✅ |

**決策**：支援 Cron 和 Heartbeat，因為：
- Cron 通用性高
- Heartbeat 與現有 OpenClaw 系統整合

### 6.2 DDL

```sql
-- 公告資料表
CREATE TABLE IF NOT EXISTS announcements (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    link_id TEXT UNIQUE NOT NULL,
    department TEXT NOT NULL,
    description TEXT NOT NULL,
    gazette_no TEXT NOT NULL,
    gazette_group TEXT NOT NULL,
    publish_date DATE NOT NULL,
    url TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 檢查日誌表
CREATE TABLE IF NOT EXISTS check_logs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    check_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    status TEXT NOT NULL,  -- 'success', 'partial', 'failed'
    crawled_count INTEGER DEFAULT 0,
    new_count INTEGER DEFAULT 0,
    error_message TEXT
);

-- 通知記錄表
CREATE TABLE IF NOT EXISTS notifications (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    announcement_id INTEGER NOT NULL,
    channel TEXT NOT NULL,  -- 'telegram', 'email', 'log'
    status TEXT NOT NULL,   -- 'pending', 'sent', 'failed'
    sent_at TIMESTAMP,
    error_message TEXT,
    FOREIGN KEY (announcement_id) REFERENCES announcements(id)
);

-- 索引
CREATE INDEX IF NOT EXISTS idx_announcements_publish_date ON announcements(publish_date);
CREATE INDEX IF NOT EXISTS idx_announcements_department ON announcements(department);
CREATE INDEX IF NOT EXISTS idx_check_logs_check_time ON check_logs(check_time);
CREATE INDEX IF NOT EXISTS idx_notifications_announcement_id ON notifications(announcement_id);
```

## 7. 錯誤處理

### 7.1 錯誤類型

| 錯誤類型 | 描述 | 處理方式 |
|----------|------|----------|
| **網絡錯誤** | HTTP 請求失敗 | 重試 3 次，失敗則記錄日誌 |
| **解析錯誤** | HTML 結構改變 | 記錄錯誤，發送警報 |
| **資料庫錯誤** | SQLite 操作失敗 | 重試，失敗則發送警報 |
| **通知失敗** | Telegram/Email 發送失敗 | 記錄錯誤，嘗試備份通道 |

### 7.2 重試機制

```python
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=4, max=10)
)
def fetch_url(url):
    response = requests.get(url, timeout=30)
    response.raise_for_status()
    return response.text
```

## 8. 效能優化

### 8.1 優化策略

| 優化點 | 方法 | 預期效果 |
|--------|------|----------|
| **HTTP 請求** | 使用 Session 重複利用連接 | 減少 50% 請求時間 |
| **HTML 解析** | 使用 lxml 代替 html.parser | 減少 70% 解析時間 |
| **資料庫** | 使用事務批量插入 | 減少 90% 寫入時間 |
| **快取** | 快取最後一次結果 | 避免重複抓取 |

### 8.2 效能指標

| 指標 | 目標值 | 測量方法 |
|------|--------|----------|
| **抓取時間** | < 5 秒 | 從請求到解析完成 |
| **檢測時間** | < 1 秒 | 從比對到識別新項目 |
| **通知時間** | < 3 秒 | 從發送到接收 |
| **總執行時間** | < 10 秒 | 完整流程 |

## 9. 安全性考量

### 9.1 敏感資料保護

| 資料類型 | 保護方式 |
|----------|----------|
| **Telegram Bot Token** | 環境變數或加密配置文件 |
| **Email 密碼** | 環境變數或加密配置文件 |
| **資料庫檔案** | 限制檔案權限 (600) |

### 9.2 配置範例

```json
{
  "telegram": {
    "bot_token": "${TELEGRAM_BOT_TOKEN}",
    "chat_id": "${TELEGRAM_CHAT_ID}"
  },
  "email": {
    "smtp_server": "smtp.example.com",
    "smtp_port": 587,
    "username": "${EMAIL_USERNAME}",
    "password": "${EMAIL_PASSWORD}"
  }
}
```

### 9.3 最佳實踐

1. **不使用硬編碼密碼**：使用環境變數
2. **限制檔案權限**：配置文件 600，資料庫 600
3. **定期更新依賴**：使用 `pip-audit` 檢查漏洞
4. **日誌脫敏**：不記錄敏感資訊
